# Agent Policy

## 角色

| 角色 | 可以做 | 不可以做 |
|---|---|---|
| 人类项目负责人 | 审核产品目标/范围；决定用户价值、科研可信度、数据安全、成本、长期形态和重大事项；批准核心架构、重要 RFC、破坏性操作和进入下一 Wave | 逐项审核代码、测试、Git、worktree、CI、接口一致性 |
| 主控 Agent | 创建直接子 Agent、拆任务、分配 ownership、组织技术验收、集成、复跑测试、核对证据、形成验收摘要、更新状态 | 创建下级层级之外的 Agent；绕过人类对产品重大事项的批准；隐瞒技术失败或证据缺口 |
| 子 Agent | 执行一个任务、修改声明路径、运行局部测试、提交交接证据 | 创建子 Agent、合并代码、修改公共契约、修改迁移、删除未知文件、宣称未经复核的最终验收 |
| QA Agent | 独立审查 diff、运行质量门槛、复核测试/CI/安全/契约证据、报告例外 | 自行改变被测实现；替代人类做产品方向决策 |

## 强制规则

- 只有主控 Agent 可以创建子 Agent；子 Agent 不得创建下级 Agent。
- 一个子 Agent 只执行一个可验收任务。
- 同一 Wave 同一文件只能有一个写入 owner。
- 子 Agent 不得自行合并、rebase 共享分支或强推远端。
- 失败时交接状态必须是 `PARTIAL` 或 `BLOCKED`，不能用“完成”掩盖阻塞。
- 主控 Agent 必须重新运行关键测试；不能把子 Agent 的输出当作验收证据。
- 主控 Agent 对技术验收结论和证据完整性负责；QA Agent 对独立质量检查结果负责。
- Wave 结束时主控必须提交人类验收摘要；人类负责人只基于摘要进行产品层面验收和下一 Wave 决定。
- Agent 发现接口或范围问题时提交 RFC，不直接修改共享契约。
