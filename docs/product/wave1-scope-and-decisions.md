# Wave 1 Product Scope and Decision Topics

状态：`CONDITIONALLY_APPROVED_NEEDS_REVISION`
用途：根据产品审核意见修订后的范围草案；仍需最终审核，不授权创建实现任务、分支或 worktree。

一页式摘要见：[wave1-product-scope-summary.md](D:/Research%20Reading/docs/product/wave1-product-scope-summary.md)

## 一、已决定的产品范围

### 1. 产品定位

- 面向个人研究者。
- 形态为单用户 Web 工作台。
- 暂不支持课题组协作、插件、本地部署和私有部署。

### 2. 阅读模式

- Wave 1 只实现“方法学习”。
- 写作支持、课题决策和实验设计只保留为未来扩展边界，不进入本轮真实实现。

### 3. 论文输入

- 允许导入一篇真实、可直接提取文本的 PDF。
- 扫描 PDF、OCR、复杂表格和公式理解明确排除。
- 建立少量固定评测论文，用于稳定验收和回归测试。

### 4. 模型使用

- 允许通过可替换的 `ModelGateway` 使用一个外部模型。
- 必须同时保留 Mock 实现，用于自动测试、失败测试和不依赖供应商的验收。
- 用户必须被告知论文内容是否发送至外部模型。
- 必须定义数据发送边界、成本上限、日志脱敏和外部服务失败处理。
- 供应商选择、适配层和具体实现由技术角色提出方案并完成技术验收；只有明显影响产品范围、隐私、成本或长期形态时才提交人类项目负责人决定。

## 二、Wave 1 最小产品闭环

Wave 1 至少覆盖以下流程：

> 创建研究项目 → 导入一篇论文 → 选择“方法学习” → 生成结构化问题计划 → 用户确认、修改或拒绝问题 → 对至少一个确认问题生成回答 → 展示原文证据片段及页码或稳定文本位置 → 用户确认、修改或拒绝回答。

边界：

- 不要求一次完成全部问题。
- 不要求高级 Evidence 检索、多篇综合、正式研究资产库或 Markdown/Excel 导出。
- 未经用户确认的回答只能是草稿，不得视为正式研究资产。

## 三、科研可信度要求

每个回答必须明确区分：

- `PAPER_FACT`：论文中可以核验的事实；
- `AUTHOR_CLAIM`：作者的主张或解释；
- `AGENT_INFERENCE`：Agent 基于证据作出的推断。

证据回答必须能够回溯到：

- 论文文件版本；
- 页码或稳定文本位置；
- 支撑回答的原文片段。

缺少可回溯证据的事实性回答不能通过产品验收。

## 四、产品验收标准

### 必须满足

1. 单用户可以完成上述最小产品闭环。
2. 只支持“方法学习”，其他阅读模式不会误进入真实流程。
3. 真实可提取文本 PDF 可以被识别、导入并关联文件版本。
4. Mock ModelGateway 和外部模型适配路径具有相同的结构化输出契约。
5. 至少一个确认问题能够生成回答草稿。
6. 回答展示证据片段以及页码或稳定文本位置。
7. 用户可以修改、拒绝问题计划和回答。
8. 回答区分论文事实、作者主张和 Agent 推断。
9. 未确认回答不会进入正式研究资产。
10. 固定评测论文能够稳定复现关键验收场景。

### 主要产品指标

- 问题可用率；
- 带证据回答的用户接受率。

### 辅助指标

- 问题修改率；
- 回答拒绝原因；
- 任务完成时间；
- 用户是否认为该流程比普通摘要更有帮助。

## 五、明确排除范围

- 写作支持、课题决策和实验设计的真实实现。
- 扫描 PDF、OCR、复杂表格和公式理解。
- 高级 Evidence 检索、多篇文献综合。
- 正式研究资产库、Markdown/Excel 导出。
- 多用户协作、课题组功能、权限体系。
- Zotero、Word、LaTeX、浏览器插件。
- 本地部署、离线产品、本地模型和私有部署。
- 自动判断创新性、研究空白或论文质量。
- 未经再次审核的额外产品模块。

## 六、仍需技术论证的事项

以下事项由主控 Agent、QA Agent 和相关责任 Agent提出方案、验证和技术结论；不默认提交人类项目负责人逐项审核：

- 前端/后端运行时与技术栈。
- 外部模型供应商候选、ModelGateway 适配层和 Mock 策略。
- PDF 文本提取、文件版本、稳定文本位置和不支持格式的识别策略。
- 数据发送边界、日志脱敏、成本计量、预算阈值和外部服务失败处理。
- 数据库、对象存储和异步任务是否需要进入 Wave 1。
- API、错误模型、任务状态、答案类型和证据 Schema。
- 固定评测论文的选择、人工基准和问题/回答评测方法。
- formatter、lint、typecheck、依赖和安全工具的 CI 接入。
- QA 的实际执行方式：独立 QA Agent，或与实现 Agent 分离的独立审查任务。

只有当这些技术方案明显改变产品范围、隐私、成本或长期产品形态时，才需要提交人类项目负责人决定。

## 七、仍需人类项目负责人决定的事项

1. 是否最终批准“个人单用户 Web 工作台”作为 Wave 1 产品形态。
2. 是否最终批准真实 PDF 和外部模型进入 Wave 1。
3. 是否认可“问题可用率”和“带证据回答的用户接受率”为主要产品指标。
4. 外部模型数据发送和成本边界中，哪些风险可以接受，哪些必须禁止。
5. 是否接受 Wave 1 只实现方法学习，其他模式延期。
6. 是否接受本轮不实现正式资产库、导出、多篇综合和高级 Evidence 检索。

## 八、开始条件

在以下条件完成前，不得创建 Wave 1 真实业务实现任务：

- 人类项目负责人最终确认本文件和一页式摘要中的产品范围。
- 主控 Agent 将批准范围转化为 Wave 计划，并建立 ownership、契约版本、验收标准和回滚方案。
- 技术角色完成 ModelGateway、PDF 输入、证据回溯、成本/隐私和评测方案论证。
- QA 执行方式已明确，并写入 Wave 计划和任务验收标准。
