# Wave 0 契约变更演练：`/health` 响应新增字段

> 本文件是多 Agent 流程演练记录，不是生产业务设计。演练假设需要为 Wave 0 的 `/health` 响应新增一个字段；本次任务不修改 `packages/contracts`，也不修改任何其他文件。

## 为什么必须提交 RFC

即使新增字段看起来是向后兼容的扩展，`/health` 仍是对外可见的共享接口。它会改变响应契约、字段语义、版本兼容承诺和验收标准，属于公共接口/共享 Schema 变更。按照仓库治理规则，公共接口和共享 Schema 不能由单个任务直接决定，必须先提交 RFC，由人类项目负责人确认字段名称、类型、是否必填、默认行为、兼容策略、范围和回滚方案。

RFC 获批前只能记录假设和测试计划；不能把假设落实到 `packages/contracts`、服务实现或消费者代码中。本次文档本身只用于验证“发现契约变更后停下来走 RFC”的流程。

## 可能受影响的消费者

- 健康检查客户端、启动探针和运维脚本：可能使用严格字段集合、响应校验或 JSON 路径读取。
- Web/API 调用方及其类型生成代码：可能需要更新响应类型、快照或展示逻辑。
- 监控、告警和日志采集：可能依赖响应字段、序列化格式或成功判定条件。
- 契约测试、集成测试、CI gate 和本地占位检查：可能比较完整 JSON、Schema 或固定快照。
- 文档、示例、SDK/测试夹具及未来的其他服务：可能把当前响应形状当作稳定接口。

具体消费者应在 RFC 阶段通过仓库搜索、ownership map、现有测试和 CI 配置确认；不能仅凭字段新增“必然无影响”作结论。

## 测试计划

RFC 获批并分配任务后，至少应验证：

1. 服务端 `/health` 成功响应包含新字段，字段类型、取值范围和语义符合获批契约。
2. 未更新的旧消费者仍能解析响应；重点覆盖严格 Schema 校验、固定快照、探针和监控脚本。
3. 已更新消费者能正确使用新字段，并覆盖字段缺失、空值、异常值和服务降级场景（若契约允许）。
4. 契约/Schema 测试、健康检查测试、相关集成测试和 CI gate 全部通过；同时执行现有 Wave 0 关键测试，避免只验证新增路径。
5. 在测试中记录命令、结果、契约版本和消费者清单，供主控 Agent 重新复跑和人工验收。

本次演练不执行真实功能测试，因为没有获批的 RFC，也没有实现或修改契约；这里只保留测试要求。

## 回滚方案

若上线后出现消费者解析失败、探针误报、监控异常或字段语义不一致，应按 RFC 中的版本和发布顺序回滚：先停止依赖新字段的消费者行为，恢复服务端原有 `/health` 响应，再回退相关消费者和测试夹具。回滚后重新执行健康检查、契约兼容性、监控/探针和 CI gate 测试，并记录事件与恢复结果。

如果新字段是可选扩展，优先让消费者忽略未知字段，并通过兼容发布降低风险；这不能替代 RFC，也不能在未经批准时擅自把字段改成可选或删除字段。

## 演练边界

本次仅新增此审计说明文件，用于演练 RFC 识别、消费者盘点、测试计划和回滚思路。明确禁止修改 `packages/contracts` 以及本 worktree 中的任何其他文件；不创建下级 Agent、不合并代码。
