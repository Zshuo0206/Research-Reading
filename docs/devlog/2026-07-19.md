## 2026-07-22 — 第四轮 managed lifecycle 收口

### 实现事实

- State durability：API、Web、Worker 创建后分别立即写入 `STARTING_API`、`STARTING_WEB`、`STARTING_WORKER`，合法 ready 后才以 `worker_ready_observed=true` 写 `READY`。写入先完成同目录 `processes.json.tmp`，已有 state 使用 .NET `File.Replace`，首次 state 使用 `Move-Item`，finally 清理 tmp；PowerShell 5.1 可用，替换失败时旧合法 state 不被半写覆盖。
- Failure classification：无 Worker 时只清理已记录且 ownership 匹配的 API/Web；Worker 已创建但从未 ready 时不要求 `CONTROL_FILE` stopped，活着则先请求 stop 并等待，已退出则直接按 pre-ready failure 收口；ready 后仍严格要求 control file、当前 Job 完成和 stopped 事件，超时保留 Worker/API/Web/state/logs。
- Crash review：ready Worker 已死且无 `CONTROL_FILE` stopped 时，start/check/stop 会进入 `CRASHED_WORKER_REVIEW_REQUIRED`。普通 stop 非零返回并保留证据；显式 `-AcknowledgeCrashedWorker` 仅在当前仓库、Worker PID 已死、ready 已观察、无 stopped 事件且 API/Web ownership 仍匹配时可用。它停止 API 再停止 Web、归档 `crashed-worker-state.json`、删除 active state，但不修改 Job、SQLite、content 或日志，也不声明 graceful stop。
- Release test isolation：生产默认入口仍固定为 `apps/worker/dist/worker.js`。临时 entrypoint 只在 `V1_LOCAL_TEST_MODE=1` 下启用，并要求绝对、存在、非仓库 `apps/**/dist`、非 Acceptance 路径；该内部能力未作为用户功能写入 runbook。Release Gate 不再读写正式 Worker dist。
- Real claimed Job test design：临时 Worker 打开独立 SQLite，使用真实 `StorageRepository`、`JobRuntime`、`runWorkerService`/`runWorkerLoop`。测试编排要求第一个 `QUESTION_PLAN` Job 实际 claim 为 RUNNING，handler 等待 release file；startup rollback 创建 `worker.stop` 后才释放 handler，Job 由真实 runtime 写为 SUCCEEDED，abort 阻止第二个 QUEUED Job 被领取，并产生 `CONTROL_FILE` stopped。另有 timeout 与 ready-after-crash/no-control 场景；这些 fixed-port 进程断言当前因 4310 listener 尚未实际进入。

## 2026-07-23 — 第五轮最终代码收口

### 实现事实

- Ready authority：`start-v1-local.ps1` 新增合法 ready 日志检测与同步 helper。当前 run 的 Worker stdout 一旦出现 `worker_platform_shell_ready`、`accepts_jobs=true` 且 readiness 包含 `database`、`migrations`、`handlers`，立即将 script-scope `worker_ready_observed` 设为 true，并在整体生命周期仍为 `STARTING_WORKER` 时原子写回 state；完整 API/Web health 与 Worker ready 条件满足后才写 `READY`。
- Observation race：`Wait-WorkerReady` 现在按 ready 日志 → startup/error 事件 → Worker ownership 存活 → API/Web health 顺序处理；startup catch 首先再次从日志同步 ready 证据。ready 已输出但 Worker 在 PowerShell 成功观察前死亡时，最终进入 `CRASHED_WORKER_REVIEW_REQUIRED`，不再误判为 ready 前失败；ready 与 error 同时出现也按 ready 后故障路径保留 state/API/Web/数据库/content/logs。
- Partial crash cleanup：`-AcknowledgeCrashedWorker` 仍要求 crash lifecycle、Worker 已死、ready 已观察、无 `CONTROL_FILE stopped`、API/Web ownership 记录存在且所有仍存活 PID 与 start time 匹配；API 或 Web 已不存在时输出 `already stopped` 并继续清理另一项。归档 state、删除 active state，不修改任何 Job 或数据库行。
- Interrupted startup：`check-v1-local.ps1` 对 `STARTING_API`、`STARTING_WEB`、`STARTING_WORKER` 会根据 ownership 记录判断；所有记录仍存活时输出 `starting`，任一停止或 PID 复用时输出 `unhealthy interrupted startup` 并返回非零。
